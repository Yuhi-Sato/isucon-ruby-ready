# ISUCON 最終チェック

## 概要

終了1時間前を目安に**新規の改善を凍結**し、計測用のオーバーヘッド除去と再起動試験に切り替える。
ここでのミスは「追試で0点」に直結する。チェックリストを上から順に、飛ばさず実施する。

再起動試験（手順3）はレギュレーションに再起動試験の実施が明記されている場合は必須（isucon-initial-recon スキル手順1で確認済みのはず）。記載が無い・不明な場合も、運営が予告なく再起動して追試することがあるため、時間が許す限り実施する。

## タイムライン目安

| 残り時間 | やること |
|---|---|
| 60分 | 改善凍結。計測系の無効化（手順1-2） |
| 45分 | 再起動試験（手順3） |
| 30分 | 最終ベンチ（手順4） |
| 以降 | **何も変更しない** |

## 手順1: 計測用オーバーヘッドを外す

すべてローカルで編集 → push → デプロイの流れで行う（サーバー直編集禁止）。

- **スロークエリログがOFFか確認**: `sN/etc/mysql/` に `slow_query_log = 0`（常時OFF運用のため通常は設定済み。MariaDBフォールバック等で有効化した場合はここで必ず戻す）
- **（任意）performance_schema無効化**: `sN/etc/mysql/` に `performance_schema = OFF`（数%のオーバーヘッド削減。以後 `make slow-query` での計測は不能になるため、最終ベンチ直前の絞り出しとしてのみ検討）
- **Vernier無効化**: `ENABLE_VERNIER` を外す（env.sh / systemdユニットを確認）。README方式で組み込んだ場合、環境変数を外せばmiddleware自体が挿入されない
- **nginxアクセスログ**: レギュレーションで再起動試験時のログ出力が問題にならなければ `access_log off;` も検討（効果は小さいことが多い。alpでの計測ができなくなるため最後に行う）
- デバッグ用の `puts` / `logger.debug` を仕込んでいたら除去

設定ファイル（`sN/` 以下）の変更は push 後、各サーバーで `git pull && make deploy-conf && make restart` で反映する（手順4の最終ベンチの `make bench` でも反映されるが、再起動試験より前に反映を済ませておくこと）。

## 手順2: 不要サービスの停止状態を永続化する

```bash
# 「止めた」だけ（stop）だと再起動で復活する。disableまで行う
sudo systemctl disable --now <不要サービス>
# 逆に、そのサーバーの役割で必要なサービスがenabledであることを確認
# （サービス名はMakefileの SERVICE_NAME / DB_SERVICE_NAME に合わせる）
systemctl is-enabled nginx <DB_SERVICE_NAME> <SERVICE_NAME>
```

## 手順3: 再起動試験

追試（運営による再起動後の動作確認）を模擬する。起動順の問題を検出するため**全サーバーを同時に**再起動する:

```bash
sudo reboot   # 全サーバーで同時に実行
# 再接続後、各サーバーの役割に応じて確認:
systemctl status <そのサーバーで動くべきサービス>   # active (running) か
# nginxが動くサーバー（s1等）でのみ:
curl -s localhost/ -o /dev/null -w '%{http_code}\n'  # トップページが200か
# DB専用サーバー（s2等）では:
mysqladmin ping   # または systemctl status <DB_SERVICE_NAME>
```

確認ポイント:

- アプリがDBより先に起動して接続失敗していないか。失敗していればsystemdユニットに `After=<DB_SERVICE_NAME>.service` / `Restart=always` を追加する（systemdユニットの編集方法・管理対象外である点は isucon-server-tuning スキル「アプリサーバー」節を参照）
- 複数台構成の場合、s2(DB)より先にs1(アプリ)が起動しても最終的に復旧するか
- 再起動後にデータが残っているか（tmpfsやメモリ上のデータに依存していないか）

## 手順4: 最終ベンチ

```bash
make bench   # ベンチの「準備」（設定反映＋全再起動＋ログ消去）
# → ベンチマーカー本体はポータル等から実行し、スコアを記録
```

- スコアが直前より大幅に下がったら、手順1の変更（特にnginx設定）を疑い、直近のdiffを確認する
- **終了間際のベンチは1回で決める余裕を残す**（ベンチキューが混む大会が多い）

## 手順5: 提出前の最終確認

- レギュレーションの提出要件（対象サーバーの申告など）を再読
- git status が全サーバーでクリーンか（`git pull` 済み・直編集の残骸なし）
- チームメンバー全員に「以降変更しない」を宣言

## よくある失敗

| 失敗 | 対策 |
|---|---|
| フォールバックで有効化したslow_query_logを戻し忘れ、無駄なI/Oでスコアを落とす | 手順1を60分前に必ず実施 |
| `systemctl stop` だけで再起動後に不要サービスが復活しメモリを圧迫 | `disable --now` まで行う（手順2） |
| 再起動試験をせず、追試でアプリが起動せず0点 | 手順3を全サーバーで実施 |
| 終了5分前に「最後の改善」を入れてベンチが回せず未検証のまま提出 | 60分前に改善凍結。凍結後はチューニングしない |
| 1台だけ再起動試験して複数台構成の起動順問題を見逃す | 全サーバーで同時に再起動して確認 |
